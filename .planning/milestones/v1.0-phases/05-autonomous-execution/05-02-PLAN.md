<task type="auto">
  <name>Task 2: Add auto-fix loop, failure escalation, and irreversible gates to execute-phase SKILL.md</name>
  <files>skills/execute-phase/SKILL.md</files>
  <action>
  Modify execute-phase SKILL.md to add four new capabilities. All changes are additive -- existing steps are preserved, new steps/modifications are inserted.

  **Modification 1: Make verification unconditional (AUTO-04)**

  In the `verify_phase_goal` step, the verification currently runs as-is. Add a note at the top of that step making it explicit that verification runs after EVERY phase execution, regardless of auto_advance setting. The existing step already does this, but add a comment/note to make it clear:
  ```
  **This step ALWAYS runs** — verification is unconditional, not gated by auto_advance.
  ```
  This is already the behavior (the step has no auto_advance check), but the explicit note prevents future regressions.

  **Modification 2: Add `auto_fix_loop` step (AUTO-05, AUTO-06, AUTO-07)**

  Insert a new step `<step name="auto_fix_loop">` AFTER `execute_waves` step 4 (spot-check) and BEFORE step 7 (next wave). This replaces the manual "ask user" failure handler in step 5 when auto_advance is active.

  The step should contain:

  1. **Entry condition:** A plan execution failed (spot-check failure or agent crash) AND auto_advance is active (either `_auto_chain_active` or `auto_advance` config is true). If auto_advance is NOT active, fall through to existing manual failure handling (step 5).

  2. **FAILURES.md initialization:** On first failure, create `.planning/FAILURES.md` using the template from `templates/failures.md`.

  3. **Retry loop (max 3 attempts per failure):**

     For attempt = 1 to 3:

     a. Write/update failure entry in FAILURES.md (orchestrator writes, NEVER agents -- per research pitfall 5).

     b. Spawn gsdr-debugger for diagnosis using `find_root_cause_only` goal:
        ```
        Task(
          prompt="<objective>Diagnose failure in plan {plan_id}</objective>
            <symptoms>
            expected: Plan executes successfully
            actual: {error_output_last_50_lines}
            errors: {error_signature}
            </symptoms>
            <mode>
            symptoms_prefilled: true
            goal: find_root_cause_only
            </mode>",
          subagent_type="gsdr-debugger",
          model="{executor_model}"
        )
        ```
        Per user decision: "spawn a debug agent first to diagnose the failure, then re-run with diagnostic context (not blind re-run)".

     c. Re-spawn executor with diagnostic context:
        ```
        Task(
          prompt="Execute plan {plan_id}. PREVIOUS FAILURE CONTEXT:
            Root cause: {diagnosis.root_cause}
            Previous solutions that DID NOT WORK: {list from FAILURES.md}
            DO NOT repeat these approaches.
            Read: .planning/FAILURES.md for full attempt history.
            [full executor prompt from execute_waves step 2]",
          subagent_type="gsdr-executor",
          model="{executor_model}"
        )
        ```
        Per user decision: "track what was tried across attempts; explicitly instruct the next attempt to NOT repeat previous solutions".

     d. Check result:
        - **Success (spot-check passes)?** Update FAILURES.md status to "resolved", break loop, continue execution.
        - **Compare with previous attempt:** Normalize error signatures. Check both signals:
          - Same normalized error signature? Strike.
          - No test improvement (result is 'no_improvement')? Strike.
          - Per user decision: "same error signature OR no test improvement -- either counts as a strike"
        - **If strike_count >= 2:** Halt this fix path. Update FAILURES.md status to "halted" with reason. Break loop.
        - **Otherwise:** Update FAILURES.md with new attempt, continue loop.

  4. **After loop exits (resolved or halted):**

     - **If resolved:** Continue to next wave normally.
     - **If halted:** Add plan to `halted_plans` set. Before spawning each subsequent plan, check if any of its `depends_on` entries are in the halted set. If yes: skip plan, write "skipped_upstream_failure" entry to FAILURES.md with `blocked_by` field. If no: execute normally.
     - Per user decision: "After two-strike halt: queue the failed task and continue executing independent tasks -- don't stop everything."

  5. **Deferred task handling at end-of-run:** After all waves complete, check FAILURES.md for entries with `status: skipped_upstream_failure`. For each: check if their `blocked_by` plan was resolved during execution. If yes, attempt execution (one try, no retry loop). If still halted, leave as skipped. Per research pitfall 3: "Before retrying deferred tasks at end-of-run, check if their upstream dependency was resolved in FAILURES.md."

  **Modification 3: Add irreversible action gate (AUTO-09)**

  In `execute_waves` step 2, BEFORE spawning the executor agent, add an irreversible action check:

  1. Read plan file content.
  2. Extract all `<action>` blocks.
  3. Load default patterns from `references/irreversible-actions.md`.
  4. Load config overrides from `.planning/config.json` key `irreversible_actions` (if exists).
  5. Load pre-approved actions from `.planning/config.json` key `pre_approved_actions` (if exists).
  6. Match patterns against action block content (not comments, not test strings).
  7. Filter out pre-approved matches.
  8. If ANY non-pre-approved match found:
     - Present gate to user with matched commands listed, REGARDLESS of auto_advance setting.
     - Per user decision: "irreversible actions ALWAYS prompt regardless of autonomy mode (AUTO-09)."
     - Wait for user confirmation before spawning executor.
     - Log pre-approved actions that were bypassed in end-of-run report for auditability.
  9. If no matches: proceed normally.

  **Modification 4: Add end-of-run report generation (AUTO-08)**

  In the `offer_next` step, when the auto-advance chain completes the final phase (is_last_phase = true) OR when cross-phase mode (--milestone/--all) finishes all batches:

  1. Generate end-of-run report by reading all SUMMARY.md, VERIFICATION.md, and FAILURES.md files.
  2. Write to `.planning/END-OF-RUN-REPORT.md`.
  3. Display summary to user.
  4. Commit the report.

  Only generate report when:
  - Auto-advance chain reaches last phase, OR
  - Cross-phase --milestone/--all completes all batches
  NOT for single manual phase execution (phase completion output is sufficient).

  **Modification 5: Update cross-phase failure handling**

  In `execute_milestone` step 2e (batch completion check), replace the "offer Retry?" interactive prompt with auto-fix loop routing when auto_advance is active. The same auto_fix_loop logic applies: diagnose, retry, halt if non-improving, continue independent phases.

  **Keep context lean:** The auto-fix loop references FAILURES.md file path (on disk) rather than inlining diagnostic content into the orchestrator prompt. Per research pitfall 1: "Keep diagnostic context in FAILURES.md (on disk), not in orchestrator memory."
